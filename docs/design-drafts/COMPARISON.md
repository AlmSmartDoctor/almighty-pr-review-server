# Almighty PR Review — 디자인 방향 비교

> **관점:** solo 파워유저가 멀티벤더(Claude + Codex) PR findings 를 **빠르게 트리아지**하고, 승인 → GitHub 포스팅 / 기각하는 로컬 단일사용자 도구.
> 세 초안은 **동일한 목업 데이터셋**(4 PR · 12 findings · CRITICAL~LOW · 합의/단일 벤더 혼재)을 공유하므로 UX 차이만 비교하면 됩니다.
> 최종 결정은 사용자가 합니다. 아래는 근거 있는 추천까지만.

| | A. Cockpit (Ops 콘솔) | B. Focus (집중 플로우) | C. Kanban (플로우 보드) |
|---|---|---|---|
| 파일 | `variant-cockpit.html` | `variant-focus.html` | `variant-kanban.html` |
| 컬러/무드 | Dark · dense | Light · calm | Light · flow |
| 은유 | Bloomberg 터미널 / 관제탑 | Superhuman 인박스 | Trello / Linear 보드 |
| 한 화면 정보량 | 매우 높음 (전역 조망) | 낮음 (finding 1개) | 중간 (카드 조망) |
| 주 입력 | 키보드 (j/k/a/x) | 키보드 + 큰 버튼 | 드래그 + 버튼 |
| 상태 모델 | 테이블 행 상태 플래그 | 큐 소비(처리/미처리) | 컬럼 = 파이프라인 단계 |

## Nielsen 휴리스틱 + 트리아지 속도 + 학습곡선 비교

점수는 이 도구의 목표(solo·고속·로컬) 기준 상대평가. ●●●=강함 · ●●○=보통 · ●○○=약함.

| 기준 | A. Cockpit | B. Focus | C. Kanban |
|---|:---:|:---:|:---:|
| **가시성**(시스템 상태) — 큐 깊이·rate-limit·리뷰 진행 | ●●● 상단 타일로 전역 상시 노출 | ●●○ 진행률 바 위주, 전역 조망 약함 | ●●● 컬럼 카운트·WIP 로 흐름 가시화 |
| **실세계 일치** — 멘탈모델 | ●●○ 관제 콘솔에 익숙해야 | ●●● "받은편지함 비우기"로 직관적 | ●●● "칸반 이동"으로 직관적 |
| **사용자 제어/자유** — 되돌리기·이동 | ●●○ 상태 변경 되돌리기 없음(초안) | ●●● undo·이전(←) 명시 지원 | ●●● 아무 컬럼으로나 재이동 자유 |
| **일관성/표준** | ●●● 벤더·심각도 배지 규약 통일 | ●●● 동일 규약을 여백에 풀어 설명 | ●●● 동일 규약 + 컬럼 색 |
| **오류 예방** | ●●○ 빠른 키가 오조작 위험 | ●●● 한 번에 하나 → 오조작 최소 | ●●○ 드래그 오투하 가능 |

## 방향별 강점 / 약점

### A. Cockpit — Ops 콘솔 (dense/dark)
- **강점**
  - findings 가 쌓였을 때(여러 레포 폴링) **처리량 최강**. j/k 로 훑고 a/x 로 즉결.
  - rate-limit 예산·큐·리뷰 상태를 **한눈에** — 이 프로젝트의 핵심 리스크(단일 시트 rate limit)를 상시 감시.
  - 심각도·벤더·합의·위치를 한 행에 압축 → **컨텍스트 스위칭 최소**.
- **약점**
  - 학습곡선이 가파르고, 빠른 키는 **오조작 위험**(현재 초안엔 undo 없음).
  - 밀도가 높아 **신중한 판단**(모호한 CRITICAL 하나를 깊게 보기)에는 부담.
  - 모바일/작은 화면 부적합.

### B. Focus — 집중 플로우 (calm/light)
- **강점**
  - **오류 예방·복구**가 구조적으로 강함(한 번에 하나 + undo). 잘못 승인이 곧장 포스팅되는 도구에선 큰 안전마진.
  - diff·설명을 **크고 읽기 좋게** → 애매한 finding 판단 품질↑.
  - 학습 불필요, 큐를 "비우는" 만족감 → **꾸준한 트리아지 습관** 유도.
- **약점**
  - **전역 조망 부재** — 지금 몇 건 밀렸는지, 어느 레포가 급한지 파악이 약함.
  - findings 가 많을 때 **처리량 병목**(한 건씩).
  - 여러 finding 을 **비교**하며 판단하기 어려움.

### C. Kanban — 플로우 보드 (flow)
- **강점**
  - **파이프라인 가시화** — "포스팅 대기(approved)" 병목, Triage 잔량, WIP 초과를 눈으로.
  - 트리아지와 실제 포스팅을 **분리**(승인=대기열, 이후 일괄 포스팅)하기 좋은 상태모델.
  - finding별/PR별 그룹핑으로 **관점 전환** 유연.
- **약점**
  - 카드 이동 = **1건당 조작 비용**. 대량 고속 처리엔 콘솔보다 느림.
  - 화면당 정보가 콘솔보다 적어 **드릴다운**(diff 상세)이 별도 필요.
  - 컬럼이 늘면 가로 스크롤·복잡도↑.

## 블렌드 추천 (어디서 무엇을 가져올까)

세 방향은 배타적이지 않습니다. 목표(solo·고속·안전한 포스팅)에 맞춰 다음 조합을 권합니다. **결정은 사용자 몫**이며, 아래는 근거 있는 옵션입니다.

1. **골격은 A(Cockpit)** — 전역 조망 + 키보드 처리량은 이 도구의 정체성. 특히 rate-limit/큐 타일은 프로젝트 리스크상 필수. 기본 작업 화면으로.
2. **A 안에 B(Focus)의 안전장치를 이식** — 선택된 finding 을 **넓게 여는 "집중 모드"**(같은 화면 우측 또는 오버레이) + **undo/실행취소**를 반드시 도입. "빠른 키 + 되돌릴 수 있음"이 고속과 안전을 동시에 잡는 핵심.
3. **상태 모델은 C(Kanban)의 파이프라인을 차용** — 승인을 곧바로 포스팅하지 말고 **"Approved(포스팅 대기)" 단계**를 두고, 스펙의 `<!-- almighty-review -->` 마커와 함께 **일괄 승인·포스팅**. WIP/대기 카운트는 A의 타일로 노출하면 별도 보드 없이도 흐름 가시성 확보.
4. **온보딩/저부하 시나리오엔 B를 서브뷰로** — findings 가 적거나 "천천히 훑고 싶은 날"엔 B의 원-카드 큐를 토글로 제공. 같은 데이터·같은 액션이라 전환 비용 낮음.

### 결정 축 (사용자가 고를 때 스스로 물을 질문)
- 하루 findings 가 **수십~수백 건**인가? → A 비중↑. **십수 건 이하 + 신중함** 우선? → B 비중↑.
- 승인과 실제 GitHub 포스팅을 **분리(검토 후 일괄)** 하고 싶은가? → C의 파이프라인 채택.
- **오조작 → 즉시 포스팅** 리스크를 얼마나 감수하나? 낮게 잡을수록 B의 undo·확인 단계 비중↑.
